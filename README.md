# landlock-helper

A CLI tool for generating Kubernetes patches and executing binaries with **Landlock** filesystem sandboxing.

`landlock-helper` helps you secure container workloads by restricting filesystem access at the kernel level using Linux Landlock LSM. It provides an end-to-end workflow: defining policies declaratively in YAML, generating Kubernetes Deployment patches/ConfigMaps, and enforcing the sandbox at runtime inside your containers.

## Installation

You can install the latest version via `go install`:

```bash
go install github.com/idefav/landlock-helper@latest
```

Or download the pre-compiled binaries from the [Releases](https://github.com/idefav/landlock-helper/releases) page.

## Commands Overview

`landlock-helper` provides three main subcommands:

1. **`generate`**: Reads your declarative YAML policy files, and outputs Kubernetes patch files (Deployment patches and ConfigMaps) that inject the sandbox into your Pods.
2. **`check`**: Verifies if the previously generated Kubernetes patches are up-to-date with your current YAML policies. Useful for CI pipelines.
3. **`exec`**: The runtime wrapper. It reads the generated `runtime.json` policy inside the container, activates the Landlock sandbox rules, and then `exec`s into your actual application binary.

## Detailed Usage & Parameters

### 1. `landlock-helper generate`

Generates Kubernetes manifest patches based on your source YAML policies.

**Usage:**
```bash
landlock-helper generate -root <directory> -index <index_file>
```

**Parameters:**
- `-root` (default: `.`): The root directory of your repository. All file paths in the index are resolved relative to this root. Generated output files will also be written relative to this root.
- `-index` (default: `configs/filesystem-policy.yaml`): The path to the index YAML file that lists all the individual container policy files.

### 2. `landlock-helper check`

Checks if the generated output files exactly match what `generate` would produce. Returns a non-zero exit code if the generated files are stale.

**Usage:**
```bash
landlock-helper check -root <directory> -index <index_file>
```

**Parameters:**
- `-root` (default: `.`): The root directory of your repository.
- `-index` (default: `configs/filesystem-policy.yaml`): The path to the index YAML file.

### 3. `landlock-helper exec`

Runs inside the target container. It reads the Landlock rules from the mounted ConfigMap (`runtime.json`), restricts the current process using `landlock_restrict_self`, and then executes your target command. The target command and all its future child processes will permanently inherit these filesystem restrictions.

**Usage:**
```bash
landlock-helper exec [-config <path>] [-container <name>] -- <your_command> [args...]
```

**Parameters:**
- `-config` (default: `$LANDLOCK_HELPER_CONFIG` or `/etc/landlock-helper/runtime.json`): The path to the runtime JSON policy file generated by the `generate` command.
- `-container` (default: `$LANDLOCK_HELPER_CONTAINER`): The name of the container. This tells the helper which specific ruleset to pick from the `runtime.json` file.
- `--`: Separator. Everything after `--` is the actual command and arguments you want to run under the sandbox.

## Configuration Files

The policy generation relies on an Index File and one or more Policy Files.

### 1. Index File

The index file (e.g., `configs/filesystem-policy.yaml`) is the entrypoint that lists all individual policy files you want to process.

```yaml
version: 1
configs:
  - name: my-app-policy
    file: configs/filesystem-policy/my-app.yaml
  - name: background-worker-policy
    file: configs/filesystem-policy/worker.yaml
```

### 2. Policy File

A policy file describes the target Kubernetes Deployment and the specific filesystem rules for each container inside that Deployment's Pod.

```yaml
version: 1
name: my-app-policy
output: k8s/overlays/prod/patch-landlock.yaml  # Where the deployment patch is written

runtime_config:
  output: k8s/overlays/prod/configmap-landlock.yaml # Where the ConfigMap is written
  config_map_name: my-app-landlock-config           # Name of the generated ConfigMap
  mount_path: /etc/landlock-helper                  # Where it will be mounted in the pod

target:
  api_version: apps/v1
  kind: Deployment
  name: my-app
  namespace: default

pod:
  containers:
    - name: app-server
      landlock:
        enabled: true
        # 'entrypoint' mode injects `landlock-helper exec --` automatically into the pod spec's command.
        # 'manual' mode means you will call `landlock-helper exec` inside your own entrypoint.sh script.
        inject_mode: entrypoint 
        compatibility: hard_requirement # 'hard_requirement' (fail if landlock unavailable) or 'best_effort'
        entrypoint:
          - /app/bin/server
          - --port=8080
        read_only_paths:
          - /            # Protect the entire root filesystem (read-only)
          - /proc        # Protect /proc
        read_write_paths:
          - /tmp         # Allow writing to temp
          - /app/data    # Allow writing to specific app data dir
          - /dev         # Allow access to device nodes (like /dev/null)
```

### 3. Runtime JSON (Generated)

When you run `landlock-helper generate`, it creates a `patch-landlock.yaml` (to modify the Deployment's command and inject the ConfigMap volume) and a `configmap-landlock.yaml`. 

The ConfigMap contains a `runtime.json` file that looks like this:

```json
{
  "version": 1,
  "containers": {
    "app-server": {
      "enabled": true,
      "compatibility": "hard_requirement",
      "include_workdir": false,
      "read_only_paths": [
        "/",
        "/proc"
      ],
      "read_write_paths": [
        "/app/data",
        "/dev",
        "/tmp"
      ]
    }
  }
}
```

At runtime, the `landlock-helper exec` binary reads this JSON, applies the Landlock rules for the `app-server` container, and then executes `/app/bin/server`.
